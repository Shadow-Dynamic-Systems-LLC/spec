# Evidence Standards

Evidence is a produced artifact that proves a control or guarantee was honored. Claims are not evidence. This document defines acceptable evidence forms per artifact type and tier.

---

## Evidence Forms

| Form | Description | Machine-verifiable |
|---|---|---|
| **Hash** | SHA-256 of a canonical output, bound to a capability invocation | Yes |
| **Test output** | Structured test result (pass/fail/coverage) with timestamp | Yes |
| **Signed record** | Cryptographically signed log entry with actor, action, timestamp | Yes |
| **Approval token** | Token issued by `human_approval_gate` on approval or rejection | Yes |
| **Audit log entry** | Append-only record emitted by `audit_log` primitive | Yes |
| **Review artifact** | Timestamped AI persona review pass with checklist results | Partial |
| **Manual attestation** | Human-signed statement for non-automatable evidence | No |

Manual attestation is permitted only where no machine-verifiable form exists. It must identify the attesting party, the control being attested, and the date.

---

## Evidence Requirements by Artifact

### `controls.md`

Each `<control>` block must specify:
- `enforcement_level`: `hard` (blocks execution) | `soft` (warns) | `advisory` (informational)
- `evidence_type`: which evidence form satisfies this control
- `evidence_produced_by`: which capability or tool produces the evidence

### `validation.md`

Evidence requirements are organized into three gates:
- **Pre-approval** — required before a governed-tier change is approved
- **Pre-merge** — required before the change branch is merged
- **Pre-release** — required before deployment to a Lighthouse execution envelope

Each gate item must reference a control ID (`CTL-NNN`) and specify the required evidence form.

### `traceability.md`

The `evidence` section records which evidence artifacts have been produced and links them to their originating controls. This section is generated by `/sds:traceability-check` — do not author manually.

---

## Tier-Specific Evidence Requirements

### Lean Tier
No formal evidence requirements. Controls and validation artifacts are not required.

### Standard Tier
No formal evidence requirements. Validation warnings are advisory.

### Brownfield Tier
Brownfield discovery evidence required:
- Dependency map produced by investigation (acceptable: structured markdown with explicit unknowns marked)
- Risk mapping with at least one identified failure mode per external dependency

### Governed Tier
All of the following are required before approval:
- `controls.md` with enforcement levels and evidence types specified for every `<control>`
- `validation.md` with all three gates populated
- `traceability.md` generated by `/sds:traceability-check` with no unlinked requirements
- `simulation.md` with no open CRITICAL findings
- Signed review artifacts from Governor and Verifier AI persona passes (context-isolated)

---

## Evidence Lifecycle

1. Evidence is declared in `controls.md` (what form is required) and `validation.md` (when it must exist).
2. Evidence is produced at runtime by capabilities (e.g., `audit_log`, `human_approval_gate`).
3. Evidence provenance is recorded in `traceability.md` under the `evidence` section.
4. `/sds:check-drift` verifies that runtime evidence matches declared guarantees in `capability.yaml`.

**Memoria routing is a declared future integration point, not yet operational.** Until the Memoria binding is specified and implemented, evidence artifacts are retained locally within the change directory and referenced by file path in `traceability.md`. The `<evidence>` tag schema is forward-compatible — provenance links authored now will be valid when Memoria routing is enabled. See SPEC.md §Future Lighthouse Binding for the declared interface.
